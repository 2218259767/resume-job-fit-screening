# 岗位匹配方法论

## 目标

把“招聘页抓取 + 简历匹配 + 严格筛选 + Markdown 交付”变成可复用流程，而不是一次性人工判断。

## 四层结构

1. `数据接入层`
   - 先把职位列表和详情稳定拿下来。
2. `规则初筛层`
   - 先排掉明显不该投的岗位。
3. `严格复筛层`
   - 对保留下来的岗位做更苛刻的独立评审。
4. `结果交付层`
   - 输出用户可以直接用于投递决策的报告。

## 数据接入优先级

按下面顺序找可复用的数据获取方式：

1. 页面背后的官方接口
2. 浏览器网络请求复原
3. 前端 bundle 或 sourcemap 反推
4. 自动化浏览器
5. HTML 抓取

结论优先保存为最终可执行方法，不要保存探索过程。

## 统一岗位 Schema

至少保留这些字段：

```json
{
  "job_id": "string",
  "title": "string",
  "detail_url": "string",
  "department": ["string"],
  "locations": ["string"],
  "job_duty": "string",
  "job_requirement": "string",
  "source_site": "string",
  "captured_at": "YYYY-MM-DD",
  "raw": {}
}
```

## 候选人画像

先把简历压成可复用的结构，再做匹配：

- `基本门槛`
  - 学历、毕业时间、是否满足实习/转正条件
- `核心优势`
  - 真正可以拿来竞争的 2-5 条主线
- `明确短板`
  - 不能假装不存在的缺口
- `用户偏好`
  - 跳过条件、地域偏好、岗位类型偏好

## 初筛分层

每个岗位先归类，再做排序：

- `跳过`
  - 触发明确排除条件
- `不建议`
  - 方向明显不对口
- `备选`
  - 有一定重合，但短板明显
- `推荐`
  - 能真正吃到候选人的主线优势

每个岗位都要留一句理由，后续汇总要能回溯。

## 严格复筛

复筛的目标不是找亮点，而是减少乐观偏差。

- 对每个 JD 使用同一份严格 prompt
- 如果可以，使用独立 subagent 并行评审
- 如果不可以，就让每个 JD 在隔离上下文中单独评审
- 只依据简历事实和 JD 原文判断

重点查看：

- 硬门槛是否踩空
- 岗位核心任务是否和候选人主线一致
- 候选人的已知短板是否正好打在岗位关键要求上
- 即使能投，是否存在更优岗位导致机会成本过高

## 排名原则

最终 Top K 不要只按分数排序，还要看：

- 是否真的吃到候选人最强经历
- 初筛通过率和面试可讲性
- 是否符合用户偏好
- 与其他岗位相比的机会成本

## 交付要求

最终文档至少包含：

- 搜索范围和日期
- 用户偏好摘要
- Top K 总表
- 每个推荐岗位的详细分析
- 全量岗位的处理结论和一句话理由
